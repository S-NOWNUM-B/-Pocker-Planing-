## 2. Что считаем целевым состоянием

Целевая схема для React Router Framework Mode v7:

- конфигурация приложения через `react-router.config.ts`
- `appDirectory` указывает на frontend app-слой
- маршруты описаны через `routes.ts`
- каждый URL обслуживается route module
- клиентский entry использует `HydratedRouter`
- при необходимости добавлен `entry.server.tsx`
- data loading, actions, error boundaries и layout nesting живут в route modules, а не в ручной сборке роутера

Для этого проекта безопаснее начинать с **Framework Mode в SPA-режиме**:

- `ssr: false`
- без включения серверного рендера на первом шаге

Это снижает риск поломки, потому что текущий frontend уже работает как SPA поверх Vite.

---

## 3. Принцип миграции

Основной принцип: **не переписывать всё за один проход**.

Нужно мигрировать по слоям и держать каждый этап проверяемым:

- сначала инфраструктура Framework Mode
- затем базовая оболочка приложения
- потом публичные страницы
- потом auth-роуты с loader/action
- потом защищённые страницы
- затем ошибка / not found / финальная зачистка

На каждом этапе:

- сохраняется доступность текущих маршрутов
- выполняется проверка сборки
- выполняется проверка типов
- выполняется ручная навигация по ключевым сценариям

## 4. План миграции

### Этап 0. Зафиксировать поведение

Перед изменением роутинга нужно описать базовую точку отсчёта.

Что сделать:

- перечислить все текущие маршруты и их назначение
- проверить редиректы и защищённые маршруты
- зафиксировать, какие страницы используют `loader`, `action`, `useNavigate`, `useParams`, `Link`, `Navigate`
- прогнать `pnpm --filter @poker/frontend typecheck`
- прогнать `pnpm --filter @poker/frontend build`

Контрольный список:

- `/`
- `/create-room`
- `/join-room`
- `/invite/:token`
- `/about`
- `/room/:roomId`
- `/login`
- `/register`
- `/dashboard`
- `*` fallback

Критерий готовности:

- есть понятный список маршрутов и их текущего поведения
- есть рабочая baseline-сборка до миграции

---

### Этап 1. Добавить Framework Mode инфраструктуру параллельно

На этом этапе старый роутинг ещё не удаляется.

Что добавить:

- `react-router.config.ts`
- новый `appDirectory` для React Router
- `routes.ts`
- `entry.client.tsx`
- при необходимости `entry.server.tsx`

Рекомендуемая стратегия для этого проекта:

- держать React Router app-слой в `apps/frontend/src/app`
- не ломать текущую структуру `pages`, `widgets`, `features`, `entities`, `shared`
- использовать route modules как тонкий слой над существующими экранами

Почему так:

- минимальный объём перемещений
- меньше риска сломать импорты
- проще сверять старую и новую версию по одному и тому же UI-коду

Технические шаги:

- заменить Vite plugin на React Router plugin
- перенести клиентский bootstrap на `HydratedRouter`
- оставить React providers внутри нового entry
- сохранить поведение `StrictMode`

Критерий готовности:

- проект собирается с новой инфраструктурой Framework Mode
- старые экраны ещё не потеряны
- новый каркас не влияет на URL-структуру

---

### Этап 2. Создать корневой route module

Нужно выделить корень приложения как route module.

Что должно быть перенесено:

- общий layout приложения
- глобальные providers
- общие сквозные элементы интерфейса
- общий error boundary / fallback

Что важно сохранить:

- `Header`
- `Footer`
- общий `<Outlet />`
- существующее поведение авторизации в шапке

Практическая схема:

- `RootLayout` становится частью корневого route module
- провайдеры приложения остаются на уровне entry или root route
- общие обёртки не дублируются по страницам

Критерий готовности:

- главная оболочка приложения работает через Framework Mode
- все вложенные страницы рендерятся через route nesting

---

### Этап 3. Перенести публичные страницы без data logic

Сначала переносим простые маршруты, где нет loader/action зависимости.

Кандидаты:

- `/`
- `/about`
- `/create-room`
- `/join-room`
- `/invite/:token`

Что сделать:

- создать route modules для этих URL
- переиспользовать текущие page components без изменения бизнес-логики
- проверить динамические сегменты
- проверить ссылки между страницами

Почему эти маршруты первыми:

- они меньше всего завязаны на auth state
- у них минимальный риск регрессии
- они дают быструю проверку правильности route nesting и navigation

Критерий готовности:

- простые публичные страницы полностью работают через Framework Mode
- переходы не отличаются от текущего поведения

---

### Этап 4. Перенести auth-роуты

Дальше переносим страницы, где есть `loader` и `action`.

Кандидаты:

- `/login`
- `/register`
- `/dashboard`

Что перенести:

- `publicOnlyLoader`
- `authLoader`
- `loginAction`
- `registerAction`
- обработку редиректов
- выдачу ошибки формы

Что проверить особенно внимательно:

- сохранение токена после входа
- поведение при невалидных креденшелах
- редирект с `/login` и `/register` на `/dashboard` для уже авторизованного пользователя
- редирект с `/dashboard` на `/login` для неавторизованного
- корректность работы `SessionManager`

Риск этой фазы:

- здесь выше всего шанс сломать навигацию и форму, потому что loader/action поведение меняется вместе с маршрутизацией

Критерий готовности:

- весь auth-flow работает на route modules
- пользовательские сценарии не поменяли поведение

---

### Этап 5. Перенести страницу комнаты и связанные параметры

Отдельно переносим сценарий комнаты:

- `/room/:roomId`

Что проверить:

- чтение параметра `roomId`
- поведение `Navigate`, если оно используется внутри страницы
- websocket-подключение
- корректность загрузки текущего состояния комнаты

Если `RoomPage` зависит от конкретной обвязки роутера, её нужно адаптировать последней, когда уже перенесены публичные и auth-страницы.

Критерий готовности:

- основной сценарий участия в комнате работает без деградации
- URL-параметры читаются и используются так же, как раньше

---

### Этап 6. Перенести not found и обработку ошибок

После миграции основных маршрутов нужно закрыть крайние случаи.

Что сделать:

- оформить route-level error boundary
- перенести fallback для неизвестных URL
- проверить поведение при ошибках loader/action/render

Что важно:

- 404 должен выглядеть и работать предсказуемо
- ошибки не должны выбрасывать пользователя в белый экран

Критерий готовности:

- неизвестные маршруты и runtime-ошибки обрабатываются штатно

---

### Этап 7. Удалить legacy router layer

Когда весь функционал перенесён и проверен:

- удалить `createBrowserRouter`-сборку
- удалить старые `routes.tsx`, если они больше не нужны
- удалить устаревшие wrappers вокруг `RouterProvider`
- убрать лишние адаптеры между старой и новой схемой

Что не удалять слишком рано:

- сами page components
- shared UI
- entities/features logic
- session/auth helpers

Критерий готовности:

- в проекте остаётся один способ описания маршрутов
- нет дублирования маршрутизаторов

## 7. Риски и как их снижать

### Риск 1. Сломать entry point

Снижение риска:

- сначала ввести Framework Mode каркас
- не удалять старую точку входа до подтверждения новой

### Риск 2. Потерять auth redirect поведение

Снижение риска:

- мигрировать auth routes отдельно
- проверять вход, выход и повторный вход вручную

### Риск 3. Разъехать по структуре файлов

Снижение риска:

- не переносить весь Feature-Sliced Design
- использовать route modules как thin wrapper вокруг текущих страниц

### Риск 4. Сломать комнату и realtime-сценарий

Снижение риска:

- переносить комнату после базовых маршрутов
- отдельно проверить WebSocket и параметр `roomId`

Миграция считается завершённой, когда:

- редиректы и guards совпадают с текущим поведением
- сборка, типизация и линт проходят
- `RouterProvider` и ручной `createBrowserRouter` больше не нужны
- в проекте нет дублирующего роутинга

## 9. Рекомендация по первому техническому решению

Для этого репозитория наиболее безопасная стартовая конфигурация такая:

- `Framework Mode`
- `SPA without SSR` на первом шаге
- route modules поверх существующих page components
- плавный перенос loader/action logic

Это даёт переход на новую архитектуру без лишнего риска для текущей работы приложения.
